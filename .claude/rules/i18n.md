## Setup obrigatório (gatilho: primeira string PT-BR aparece)

**Estado atual:** `next-intl@4.12.0` instalado, NÃO configurado. Wire ANTES da primeira `<Button>Salvar</Button>`.

1. **`i18n/request.ts`** — config server-side:

   ```ts
   import { getRequestConfig } from 'next-intl/server'

   export default getRequestConfig(async () => {
     const locale = 'pt-BR' // fixo dia 0 (ADR-0040 §G, decisão 13)
     return {
       locale,
       messages: {
         common: (await import(`../messages/${locale}/common.json`)).default,
         // namespaces JIT via dynamic import por feature
       },
     }
   })
   ```

2. **`next.config.ts`** — plugin:

   ```ts
   import createNextIntlPlugin from 'next-intl/plugin'
   const withNextIntl = createNextIntlPlugin('./i18n/request.ts')
   export default withNextIntl(nextConfig)
   ```

3. **`app/layout.tsx`** — `NextIntlClientProvider` envolvendo `RouteProvider`:

   ```tsx
   import { NextIntlClientProvider } from 'next-intl'
   import { getMessages, getLocale } from 'next-intl/server'

   async function DynamicShell({ children }) {
     const locale = await getLocale()
     const messages = await getMessages()
     return (
       <NextIntlClientProvider messages={messages} locale={locale}>
         <RouteProvider {...routeProps}>
           <EntitlementProvider {...entProps}>
             {children}
             <Toaster richColors closeButton position="top-center" />
           </EntitlementProvider>
         </RouteProvider>
       </NextIntlClientProvider>
     )
   }

   export default function RootLayout({ children }) {
     // <html lang={locale}> dinâmico — fallback 'pt-BR' se Next 16 async layout forçar
     return <html lang="pt-BR" ...><body>...</body></html>
   }
   ```

## Como usar `t()`

**Server Components / Server Actions:**

```ts
import { getTranslations } from 'next-intl/server'

export async function MyServerComponent() {
  const t = await getTranslations('common.actions')
  return <button>{t('save')}</button>
}
```

**Client Components:**

```tsx
'use client'
import { useTranslations } from 'next-intl'

export function MyClient() {
  const t = useTranslations('common.actions')
  return <button>{t('save')}</button>
}
```

## AppError factory com i18n

`lib/contracts/errors.ts` aceita `string | I18nMessage`:

```ts
export type I18nMessage =
  | string
  | { key: string; fallback: string; metadata?: Record<string, unknown> }

// Uso server-side (ex: lib/entitlements/server.ts)
throw AppError.forbidden({
  key: 'entitlements.feature_blocked',
  fallback: 'Plan upgrade required',
  metadata: { feature: 'chatbot' },
})
```

Server loga `fallback` (EN, Sentry-friendly). Client lê `error.metadata.i18nKey` e chama `t(key)`.

Back-compat: `AppError.forbidden('msg crua')` ainda aceita string (não quebra callsites antigos).

## Zod messages — factory por callsite

**Pattern correto (Opção A — decisão 24):**

```ts
import { z } from 'zod'
import { useTranslations } from 'next-intl'

export function useEmailSchema() {
  const t = useTranslations('common.validation')
  return z.object({
    email: z.string().email({ message: t('invalid_email') }),
  })
}
```

**NÃO USAR:**

- `z.setErrorMap(...)` global em layout — hidden state, impossível debugar locale
- `.message('string crua')` — ESLint regra 11 bloqueia

**Certo:**

```json

```

**Errado:**

```json

```

## Conteúdo gerado por tenant — NÃO usa `t()` (Plano Dia 1, Decisão 2)

**Princípio:** strings escritas pelo dev (chrome do app) usam `t()`. Strings escritas
pelo profissional/IA no editor de form/landing são **dado do tenant**, ficam **inline
no JSONB** do spec. Forçar `t()` em conteúdo do tenant é confundir as camadas — gera
chaves dummy explodindo no JSON e impede o profissional de criar form sem deploy.

### Mapa de decisão

| Tipo de texto                                | Onde mora               | Quem escreve        |
| -------------------------------------------- | ----------------------- | ------------------- |
| Botão "Salvar", menu "Configurações"         | `messages/pt-BR/*.json` | Dev                 |
| Erro "E-mail inválido", "Campo obrigatório"  | `messages/pt-BR/*.json` | Dev                 |
| Label "Próxima pergunta", "3 de 7"           | `messages/pt-BR/*.json` | Dev                 |
| Pergunta "Qual seu objetivo de treino?"      | JSONB do form no banco  | Profissional / IA   |
| Opção "Hipertrofia", "Emagrecimento"         | JSONB do form           | Profissional / IA   |
| Mensagem boas-vindas e agradecimento do form | JSONB do form           | Profissional / IA   |
| Headline da landing, copy do FAQ, CTA        | JSONB da page           | Profissional / IA   |
| Body do prompt-template IA                   | DB `prompt_templates`   | Dev (versão pinada) |

### Multi-idioma de conteúdo do tenant

Quando profissional quiser form em PT-BR + EN, o spec ganha overlay:

```jsonc
{
  "locale": "pt-BR",
  "steps": [...],
  "translations": {
    "en": {
      "block-id-1": { "label": "What is your goal?" }
    }
  }
}
```

Renderer aplica overlay quando `?lang=en` na URL. Sem dependência de `messages/*`.

### Anti-patterns específicos a evitar

- Tentar gerar chave `t()` dinâmica pra label de pergunta criada pelo profissional → quebra build
- ESLint `react/jsx-no-literals` falhando em renderer de form/page → adicionar exceção de path em `eslint.config.mjs` pra `components/app-form-renderer*`, `components/app-page-renderer*`, `components/app-form-block-*`, `components/app-page-block-*`
- Schema validation rejeitando strings sem chave de tradução → Zod permite string livre nos campos `label`, `description`, `helpText`, `placeholder` etc. dos blocks

### Paths onde literal de string PT-BR/EN é PERMITIDO

```
components/app-form-renderer*.tsx        # consome JSONB do form
components/app-form-block-*.tsx          # renderiza label/description do block
components/app-page-renderer*.tsx        # consome JSONB da page
components/app-page-block-*.tsx          # renderiza props do page block
lib/forms/seed-templates/**/*.ts         # seeds dos form_templates com copy fitness embutida
lib/pages/seed-templates/**/*.ts         # seeds dos page_templates
supabase/seed/**/*.sql                   # INSERTs em form_templates/page_templates/prompt_templates
```

Adicionar ao `eslint.config.mjs` em regra `react/jsx-no-literals`:

```js
{
  files: [
    'components/app-form-renderer*.tsx',
    'components/app-form-block-*.tsx',
    'components/app-page-renderer*.tsx',
    'components/app-page-block-*.tsx',
    'lib/forms/seed-templates/**/*.ts',
    'lib/pages/seed-templates/**/*.ts',
  ],
  rules: { 'react/jsx-no-literals': 'off' },
}
```

## Anti-patterns

- String hardcoded em JSX → use `{t('key')}` (ESLint `react/jsx-no-literals` bloqueia)
- String passada por prop → use `i18nKey` no wrapper
- `aria-label="Fechar"` literal → `aria-label={t('common.actions.close')}` (ESLint bloqueia)
- `toast.success('Salvo')` literal → `useAppToast().success('common.feedback.saved')` (ESLint bloqueia)
- `new Error('msg')` literal → `throw AppError.invalidInput({ key, fallback })` (ESLint bloqueia)
- `.message('texto')` em Zod → factory `useXSchema()` (ESLint bloqueia)
- Schema `tenant_copy_overrides` dia 0 → NÃO criar
- Locale switcher UI dia 0 → NÃO criar
- Rota `app/[locale]/` dia 0 → NÃO criar

| **SEO multi-locale obrigatório** OU **2+ locales com user-switch**                                      | Migrar `app/*` pra `app/[locale]/*` + atualizar `getLocale()` + hreflang + sitemap. Custo: ~2-4h                                                 |

| **PWA offline messages preciso** (Sprint 14, ADR-0014)                                                  | Pre-cache `messages/**/*.json` via Serwist. `@serwist/next` default cobre static JSON                                                            |
| **Stripe internacional ativa cliente real com moeda ≠ BRL**                                             | Junto com locale: revisar formatadores (`Intl.NumberFormat` por locale)                                                                          |

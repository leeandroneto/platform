## 5. Audit `app/admin/theme-studio/actions.ts` + `_helpers.ts`

5 server actions auditadas: `bootstrapTenantTheme`, `saveThemeVersion`, `listThemeVersions`, `restoreThemeVersion`, `forkTheme`.

| Criterio                                                                       | actions.ts    | \_helpers.ts |
| ------------------------------------------------------------------------------ | ------------- | ------------ |
| A. Return `Result<T, AppError>` via `ok()`/`fail()`                            | OK            | OK           |
| B. JWT claims validation (`assertTenantMatch` via RPC `current_tenant_id`)     | OK            | OK (helper)  |
| C. Error handling via try/catch -> `fail()`                                    | OK            | OK           |
| D. AppError com i18n key (`{ key, fallback }`)                                 | OK            | OK           |
| E. Zod parse input (`ThemeSchema.safeParse`, `ForkThemeInputSchema.safeParse`) | OK            | n/a          |
| F. NAO usar `createClient()` direto — usa `createAdminClient()` SSOT           | OK            | OK           |
| G. Idempotent quando aplicavel (cap 50 via trigger DB; bootstrap idempotent)   | OK            | OK           |
| H. Logging server-only se complexo                                             | n/a (concise) | n/a          |

**Criterio B detalhado (CRITICO — `feedback_secdef_validates_tenant_id`):**

- `assertTenantMatch(tenantId)` em `_helpers.ts:37-45` chama RPC `current_tenant_id()` (que le JWT claim no body — SSOT), compara contra `tenantId` recebido. Se distintos -> throw `AppError.forbidden({ key: 'themes.tenant_mismatch', ... })`.
- **NAO confia no arg** — usa JWT como source of truth. OK Alinha "trust the JWT, not the args" do memory `feedback_secdef_validates_tenant_id.md`.
- `saveThemeVersion` (linha 113-123): carrega theme via admin client, **depois** chama `assertTenantMatch(theme.tenant_id)` — JWT precisa bater com tenant dono do theme. Ordem correta (nao permite enumeracao de outros tenants).
- `restoreThemeVersion` (linha 247): chama `assertTenantMatch(input.tenantId)` **antes** de operar; depois compara `themeOwner.tenant_id !== input.tenantId` (linha 269) — dupla validacao (defense-in-depth).
- `forkTheme` (linha 351): chama `assertTenantMatch(sourceTheme.tenant_id)` apos carregar theme — usa tenant do DB, nao do user input. OK

**Apenas 1 lacuna pequena (flag pra Fase 3):** `forkTheme` carrega `sourceTheme` antes de `requireEntitlement('theme_studio')` — micro-leak: usuario sem entitlement pode descobrir se `themeId` existe via 404 vs 403. Nao e cross-tenant leak (assertTenantMatch garante), apenas existencia de UUID. **Severidade baixa** — fica como FLAG (nao fix nesta fase porque e refactor cosmetico).

**Criterio D detalhado:** todos `AppError.notFound/forbidden/invalidInput/internal` em `actions.ts` usam `{ key, fallback }`. Exemplos: `themes.tenant_not_found`, `themes.snapshot_invalid`, `themes.apca.failed`, `themes.fork_invalid_input`, `themes.tenant_mismatch`. Alinha rule `i18n.md`.

**Criterio F detalhado:** `actions.ts` importa `createAdminClient` de `@/lib/supabase/admin` (nao `createClient()` raw). `_helpers.ts` importa AMBOS `createAdminClient` (writes via admin) + `createClient` (RPC `current_tenant_id` que precisa do JWT do user). Correto — adapter Supabase server SSOT, nao cliente cru.

**Resumo Audit 5:** server actions criticas estao **rigorosamente alinhadas** com rules `server-actions.md` + `jwt-claims.md` + `i18n.md`. Nenhum fix urgente. 1 micro-flag pra Fase 3 (entitlement check antes de load).

## 7. Fixes aplicados nesta fase

Lista cirurgica:

### 7.1 `lib/design/palettes.ts`

**Antes:**

```ts
throw new Error(`Palette not found: ${slug}`)
```

**Depois:**

```ts
throw AppError.notFound({
  key: 'palettes.not_found',
  fallback: `Palette not found: ${slug}`,
  metadata: { slug },
})
```

Razao: alinha rule `i18n.md` — `AppError` factory com i18n key/fallback (vs `new Error` literal). Anti-pattern documentado: `new Error('msg crua')` quebra resolucao i18n na UI.

### 7.2 `lib/design/fonts/google-fonts.ts`

**Antes:**

```ts
if (!response.ok) throw new Error(`Google Fonts API error: ${response.status}`)
// ...
console.log(`Fetched ${fonts.length} fonts from Google Fonts API`)
// ...
} catch (error) {
  console.error('Failed to fetch Google Fonts:', error)
  throw error
}
```

**Depois:**

```ts
if (!response.ok)
  throw AppError.externalService({
    key: 'fonts.google_api_error',
    fallback: `Google Fonts API error: ${response.status}`,
    metadata: { status: response.status },
  })
// ... (sem console.log)
} catch (error) {
  throw AppError.from(error)  // idempotent — preserva AppErrorImpl
}
```

Razao: alinha `AppError.externalService` (categoriza erro externo Google API corretamente); remove `console.log` (server-only side-effect; observability deve ir via Sentry/structured logging); `AppError.from()` e idempotent (retorna `AppErrorImpl` unchanged — confirmado em `lib/contracts/errors.ts:125-129`).

### 7.3 Nao aplicados (decisao consciente)

- **Hooks `'use client'`**: convencao do codebase NAO usa em `lib/hooks/*` (sheriff handles via `side:client` tag). Adicionar quebraria padrao estabelecido.
- **`new Error(\`template literals\`)`em outros`lib/design/\*`**: ESLint regra `no-restricted-syntax` bloqueia apenas `Literal` (string literal pura), nao `TemplateLiteral`. Os 5 callsites restantes (`contrast.ts` x3, `fonts/index.ts` x1, `use-font-search.ts` x1) usam template literals com interpolation, lint passa. **Refactor opcional Fase 3** (uniformizar para `AppError`) — nao bloqueia esta fase.

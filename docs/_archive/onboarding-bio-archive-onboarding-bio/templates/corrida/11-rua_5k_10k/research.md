## 1 · Quem é esse público no Brasil

### Perfil demográfico

- **Faixa etária média:** 34 anos, com rejuvenescimento acelerado (faixa 20-24 cresceu >2.300% em provas 5K/10K entre 2024-2025).
- **Gênero:** 51,8% mulheres, 48,2% homens — equilíbrio inédito, confirmado pelo estudo PeRunning 2026.
- **Socioeconômico:** Classe C representa 43% dos corredores (era 36% em 2024). Composição racial: 43% branco, 36% pardo, 10% preto. Corrida é democrática e acessível.
- **Frequência de treino média:** 1,8 sessões/semana, ~10,6 km semanais. A maioria treina 2-3x por semana — são comprometidos mas não profissionais.

### Ordem de grandeza

**Grande mercado.** O Brasil registrou 11,9 milhões de finishers oficiais entre 2023-2025, com crescimento de 85% no número de provas (2.827 → 5.241 eventos). Sudeste concentra 51,2% dos participantes, com SP sozinha hospedando 1.311 provas em 2025.

### Onde estão online

- **Strava:** 23 milhões de usuários no Brasil, crescimento de 109% em clubs de corrida. É rede social + log de treino — THE plataforma.
- **Instagram:** Assessorias de corrida dominam (ex: @goassessoriaesportiva, 43K+ seguidores). Perfis como Corrida Perfeita, O2 Corre, WebRun, Atletis são hubs de conteúdo.
- **WhatsApp:** Central — coaches comunicam via WhatsApp, grupos de corrida organizam treinos e provas ali.
- **Apps:** Strava (dominante), Nike Run Club, Garmin Connect, Polar.

### Linguagem que usam

**Vocabulário identitário do corredor de rua brasileiro:**

| Termo              | Significado                                                             |
| ------------------ | ----------------------------------------------------------------------- |
| Pace               | Tempo por km (ex: "pace 5:30" = 5min30s/km) — A métrica mais importante |
| RP                 | Recorde pessoal                                                         |
| Longão             | Corrida longa, geralmente aos fins de semana                            |
| Tiro / Intervalado | Trabalho de velocidade — repetições em alta intensidade                 |
| Rodagem            | Corrida fácil/contínua, base de volume                                  |
| Regenerativo       | Corrida muito leve, recuperação ativa                                   |
| PSE                | Percepção Subjetiva de Esforço (escala 1-10)                            |
| Parcial            | Split de tempo por km                                                   |
| Negative split     | Segunda metade mais rápida que a primeira                               |
| Assessoria         | Grupo/serviço de coaching de corrida                                    |
| Planilha           | Plano de treino                                                         |
| Prova              | Evento de corrida                                                       |
| Canelite           | Periostite tibial / shin splints                                        |

### O que ofende ou afasta

- Chamar de "recreativo" — o corredor que faz 5K em 35min se vê como corredor de rua, não como recreativo
- Jargão técnico em inglês sem tradução (VO2max, lactate threshold) — pode usar, mas precisa contextualizar
- Implicar que precisa ser rápido para ser "corredor de verdade"
- Foco excessivo em peso/corpo — muitos correm para saúde mental e comunidade, não emagrecimento
- Promessas de tempo ("quebre 25min em 5K em 4 semanas")

### Dor mais comum

**Estagnação de pace.** O corredor progride rápido no início (de 7:00/km para 6:00/km), depois trava. Corre toda sessão na mesma intensidade moderada (a "armadilha da zona 3"), não diferencia rodagem de intervalado, não periodiza. Não sabe por quê não evolui. Essa frustração leva à busca por assessoria — é o gatilho de conversão.

## 2 · Decisão de escopo

### Este template vai cobrir

- Corredor de rua que **já corre** (mínimo 1-2 meses de prática regular)
- Objetivos: baixar tempo em 5K, completar/melhorar 10K, se preparar para primeira prova cronometrada
- Faixa: desde o corredor que faz 5K em 40min até o que busca sub-20
- Com ou sem relógio/GPS (condicional)

### Este template NÃO vai cobrir

- **Iniciante absoluto** que nunca correu → Template #10 (Corrida recreativa / começar)
- **Meia-maratona / maratona** → Templates #12, #13
- **Trail / ultra** → Template #14
- **Reabilitação pós-lesão** → Template #15
- **Corredor competitivo federado** com VO2max testado em laboratório → Custom / assessoria presencial

### Decisão de naming

O label sugerido "5K / 10K (rua)" é funcional mas frio. Proponho **"Corrida de Rua — 5K / 10K"** como label no hub e `rua_5k_10k` como `specialty_code`. A palavra "rua" é identitária para esse público — eles são "corredores de rua", não "corredores de 5K". A distância (5K/10K) qualifica o template mas não define a identidade.

## 3 · Motores escolhidos

### Motores considerados e decisão

| Motor                      | Decisão                   | Justificativa                                                                                                                |
| -------------------------- | ------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| 1 — Contexto atual         | ✅ Incluído               | Preciso saber frequência e histórico REAL de treino para calibrar VDOT e tom                                                 |
| 2 — Gargalo                | ✅ Incluído               | O gargalo define o arco narrativo: quem está estagnado ≠ quem se lesiona ≠ quem não periodiza                                |
| 3 — Nível / Maturidade     | ❌ Descartado             | Nível é inferido pela combinação de contexto + pace/tempo informado. Pergunta separada seria redundante                      |
| 4 — Comportamento          | ✅ Incluído               | Frequência real de treino semanal (não intenção) define prescrição e expectativa de resultado                                |
| 5 — Ambiente               | ❌ Descartado             | Corredor de rua corre na rua — variação de ambiente (esteira, pista, rua) é marginal e não muda substancialmente o relatório |
| 6 — Identidade / Fase      | ✅ Incluído               | A distância-alvo (5K vs 10K) e se já competiu define o branch de segmentação                                                 |
| 7 — Métricas / Ferramentas | ✅ Incluído (condicional) | Se tem relógio GPS / usa Strava, posso pedir pace recente e calcular VDOT. Se não tem, uso percepção                         |
| 8 — Liberação / Risco      | ✅ Incluído               | Condições de saúde — especialmente cardíacas — são relevantes para corrida                                                   |

**Total: 5 motores → 6 perguntas específicas** (+ 1 condicional de métricas). Dentro do range 5-8.

### Lista final

1. **Segmentação (Motor 6 — Identidade/Fase)** → `race_focus` — Qual distância o corredor está focando e se já fez prova
2. **Contexto atual (Motor 1)** → `training_frequency` — Frequência real de treino semanal atual
3. **Gargalo (Motor 2)** → `biggest_barrier` — O que está travando a evolução
4. **Métricas (Motor 7)** → `uses_gps` — Se usa relógio/app GPS para treinar

6. **Safety (Motor 8)** → `health_conditions` — Condições de saúde relevantes

## 4 · Perguntas e opções

### Q1 · `race_focus` _(Motor 6 — Identidade/Fase)_

**Type:** `single_choice`
**Label (client-facing):** "Qual é seu foco na corrida agora?"
**Helper:** "Pense no que te motiva a calçar o tênis hoje"
**Required:** sim
**Visibility:** sempre
**Segmentação:** sim (abre branches)
**depthRequired:** quick

**Opções:**

- **`first_5k`** — "Quero completar minha primeira prova de 5K"
  - Safety trigger? Não

- **`improve_5k`** — "Quero baixar meu tempo nos 5K"
  - Safety trigger? Não

- **`improve_10k`** — "Quero completar ou melhorar nos 10K"
  - Safety trigger? Não

- **`first_race`** — "Nunca fiz prova, quero começar a competir"
  - Safety trigger? Não

**Justificativa da pergunta:** O hub já definiu "corrida de rua" como modalidade. Agora preciso saber se o corredor está buscando 5K ou 10K, e se já tem histórico de provas. Isso define: (a) o arco narrativo do relatório, (b) quais métricas fazem sentido, (c) qual progressão sugerir. Uma alternativa seria perguntar "qual distância?" em formato direto — descartei porque a fase importa tanto quanto a distância (quem quer completar ≠ quem quer melhorar).

**Justificativa das opções:** Cobrem os 4 perfis reais que um coach de corrida encontra nesse segmento: estreante em provas (first_5k + first_race), evoluidor de 5K (improve_5k), e evoluidor/estreante de 10K (improve_10k). Não inclui "manter forma" porque esse público não busca assessoria ativa — é target do template #10.

### Q2 · `training_frequency` _(Motor 4 — Comportamento)_

**Type:** `single_choice`
**Label (client-facing):** "Na última semana, quantas vezes você correu?"
**Helper:** "Conte só corrida, não caminhada ou academia"
**Required:** sim
**Visibility:** sempre
**Segmentação:** não
**depthRequired:** quick

**Opções:**

- **`freq_1`** — "1 vez"
  - Safety trigger? Não

- **`freq_2_3`** — "2 a 3 vezes"
  - Safety trigger? Não

- **`freq_4_5`** — "4 a 5 vezes"
  - Safety trigger? Não

- **`freq_6_plus`** — "6 vezes ou mais"
  - Safety trigger? Não

**Justificativa da pergunta:** "Na última semana" força resposta concreta, não aspiracional. Frequência REAL é a variável que mais impacta a prescrição — determina quantos tipos de sessão cabem no plano. Alternativa descartada: "quantas vezes você PRETENDE treinar?" — captura intenção, não comportamento. O prompt do PROMPT-PESQUISA é explícito: Motor 4 captura comportamento recente e concreto.

**Justificativa das opções:** 4 faixas que cobrem 95%+ dos corredores (ninguém corre 0x/sem e busca template de 5K/10K — esse é template #10). Cada faixa muda substantivamente a prescrição.

### Q3 · `biggest_barrier` _(Motor 2 — Gargalo)_

**Type:** `single_choice`
**Label (client-facing):** "O que mais está travando sua evolução na corrida?"
**Helper:** "Escolha o que pesa mais hoje"
**Required:** sim
**Visibility:** sempre
**Segmentação:** não
**depthRequired:** quick

**Opções:**

- **`pace_plateau`** — "Meu pace estagnou — não consigo ficar mais rápido"
  - Safety trigger? Não

- **`injury_fear`** — "Lesões ou dores me impedem de treinar como quero"
  - Safety trigger? Não

- **`no_structure`** — "Não sei como montar um treino — corro sem plano"
  - Safety trigger? Não

- **`time_consistency`** — "Falta de tempo ou constância — começo e paro"
  - Safety trigger? Não

- **`nutrition_doubt`** — "Não sei como me alimentar para correr melhor"
  - Safety trigger? Não

**Justificativa da pergunta:** Cada gargalo abre um arco narrativo completamente diferente no relatório. Quem está estagnado precisa ouvir sobre polarização de treino. Quem tem dor precisa ouvir sobre fortalecimento. Quem não tem plano precisa de estrutura básica. Quem não come direito precisa de contexto nutricional. A IA calibra os pilares com base nessa resposta. Alternativa descartada: múltipla escolha ("marque todos que se aplicam") — gera dados difusos e relatório genérico. Forçar 1 escolha = foco.

**Justificativa das opções:** Os 5 gargalos vêm da literatura e da prática de assessorias de corrida brasileiras. `pace_plateau` é o #1 absoluto. `injury_fear` é o #2 (joelho = 28% das lesões). `no_structure`, `time_consistency` e `nutrition_doubt` cobrem os gargalos de processo/estilo de vida. Não inclui "falta de equipamento" — corrida exige tênis e pronto.

### Q4 · `uses_gps` _(Motor 7 — Métricas/Ferramentas)_

**Type:** `single_choice`
**Label (client-facing):** "Você usa relógio ou app com GPS quando corre?"
**Helper:** "Strava, Garmin, Apple Watch, Nike Run Club ou similar"
**Required:** sim
**Visibility:** sempre
**Segmentação:** não
**depthRequired:** quick

**Opções:**

- **`yes_watch`** — "Sim, uso relógio GPS (Garmin, Apple Watch, Polar, etc.)"
  - Safety trigger? Não

- **`yes_phone`** — "Sim, uso app no celular (Strava, Nike Run Club, etc.)"
  - Safety trigger? Não

- **`no_gps`** — "Não uso nada — corro no feeling"
  - Safety trigger? Não

**Justificativa da pergunta:** Essa pergunta é o gate para Q5 (pace recente). Se não usa GPS, não faz sentido pedir pace — a pessoa não sabe. Além disso, o tipo de ferramenta (relógio vs celular vs nada) calibra a sofisticação das recomendações. Para quem tem relógio, posso recomendar treino por zona de FC. Para quem usa celular, treino por pace. Para quem não mede, treino por PSE. Alternativa descartada: assumir que todo corredor mede — dados de mercado mostram que muitos correm sem app.

### Q5 · `recent_pace` _(Motor 7 — Métricas condicional)_

**Type:** `number` (com formatação pace mm:ss/km)
**Label (client-facing):** "Qual foi seu pace médio na última corrida?"
**Helper:** "O número que aparece no app, em min/km. Ex: 5:30 significa 5 minutos e 30 segundos por km"
**Required:** não (opcional mesmo quando visível)
**Visibility:** `if uses_gps == yes_watch OR uses_gps == yes_phone`
**Segmentação:** não
**depthRequired:** standard

**Opções:** campo numérico (pace em formato mm:ss/km, range 3:00 — 10:00)

**Justificativa da pergunta:** O pace recente é a variável mais poderosa do template. Com pace + distância, calculo VDOT (Jack Daniels), que permite: (a) estimar VO2max, (b) derivar zonas de treino (Easy, Tempo, Interval, Repetition), (c) prever tempo em outras distâncias (Riegel), (d) contextualizar onde o corredor está na curva de percentis brasileiros (Pecanha et al. 2021). Sem pace, o relatório ainda funciona (usa dados demográficos + frequência), mas perde profundidade significativa.

**Alternativa descartada:** Pedir tempo de prova (ex: "qual seu tempo em 5K?"). Problema: muitos corredores nunca fizeram prova cronometrada mas sabem seu pace médio do Strava. Pace médio de treino é dado mais acessível. Ajuste: pace de treino ≠ pace de prova — aplico fator de correção (~5% mais rápido em prova) no cálculo.

### Q6 · `health_conditions` _(Motor 8 — Safety)_

**Type:** `multiple_choice`
**Label (client-facing):** "Você tem alguma dessas condições?"
**Helper:** "Marque todas que se aplicam. Isso ajuda a personalizar suas recomendações com segurança"
**Required:** sim
**Visibility:** sempre
**Segmentação:** não
**depthRequired:** quick
**maxSelections:** 4

**Opções:**

- **`cardiac`** — "Problema cardíaco (arritmia, sopro, dor no peito ao esforço)"
  - Safety trigger? **Sim** — reason: "Risco de evento cardíaco agudo durante exercício de alta intensidade. Requer ECG + teste ergométrico + liberação cardiológica antes de prescrição de intervalados/tiros."

- **`respiratory`** — "Asma ou dificuldade respiratória frequente"
  - Safety trigger? Não (asma controlada não é bloqueante)

- **`joint_injury`** — "Lesão articular ativa (joelho, tornozelo, quadril)"
  - Safety trigger? **Sim** — reason: "Risco de agravamento de lesão estrutural (menisco, ligamento, cartilagem). Corrida é impacto repetitivo — progredir volume/intensidade com lesão ativa pode causar dano irreversível."

- **`diabetes`** — "Diabetes (tipo 1 ou tipo 2)"
  - Safety trigger? Não (diabetes controlado não é bloqueante; ajusta tom e recomendações nutricionais)

- **`recent_sedentary`** — "Fui sedentário por mais de 1 ano e voltei a correr recentemente"
  - Safety trigger? Não (não é bloqueante, mas ajusta a agressividade da prescrição)

- **`none`** — "Nenhuma dessas"
  - Safety trigger? Não

**Justificativa da pergunta:** Corrida envolve impacto repetitivo e esforço cardiovascular — os dois sistemas que mais podem causar problemas agudos. As opções cobrem as condições que realmente mudam a conduta do relatório. Alternativa descartada: pergunta aberta ("tem algum problema de saúde?") — gera texto livre difícil de processar e não ativa safety triggers automáticos. Múltipla escolha permite combinações (ex: cardíaco + sedentário recente → safety reforçado).

**Justificativa das opções:** `cardiac` e `joint_injury` são safety triggers porque exigem avaliação presencial antes de prescrição de intensidade. `respiratory`, `diabetes` e `recent_sedentary` ajustam tom e recomendações mas não bloqueiam — asma controlada, diabetes medicado e retorno gradual são cenários gerenciáveis com orientação. `none` é saída limpa.

## 5 · Branches

### Branch: `Primeira Prova` (trigger: `race_focus == first_5k OR race_focus == first_race`)

- **Tom geral:** Acolhedor e confiante. O corredor está dando um passo importante — validar a coragem da decisão. Zero pressão por tempo. Foco em experiência, não em performance.
- **pillarGuidance:** "Priorize construção de hábito e progressão segura. Não mencione RP, rankings ou comparações. Use 'sua primeira largada' como âncora emocional. Fale sobre o que esperar no dia da prova: kit, largada, hidratação no percurso, medalha."
- **Additional questions:** Nenhuma
- **Remove questions:** `recent_pace` perde peso (se preenchido, usa para contextualizar, mas não é central)
- **Metrics override:** Omitir projeção de tempo competitiva. Incluir "meta de conclusão" em vez de "meta de pace". Não mostrar percentil.
- **Narrative arc override:** "Você decidiu → você já corre → a prova é o próximo passo natural → veja como se preparar nas próximas semanas → o profissional vai te guiar nessa jornada"

**Justificativa:** Quem nunca fez prova não se beneficia de VDOT, zonas de treino avançadas, ou projeções de tempo. O relatório precisa ser sobre preparação para a experiência, não otimização de performance. Mostrar "seu VDOT estimado é 32" para quem nunca pisou numa largada é irrelevante e intimidador. A mudança é substancial: pilares diferentes, métricas diferentes, tom diferente.

---

### Branch: `Evoluidor 5K` (trigger: `race_focus == improve_5k`)

- **Tom geral:** Técnico-motivacional. O corredor sabe o que é correr — quer entender POR QUÊ estagnou e COMO quebrar o platô. Pode usar conceitos como polarização, limiar, intervalado. Celebrar RP como conquista legítima.
- **pillarGuidance:** "Foque em polarização de treino (80/20), zonas de intensidade, e trabalho de velocidade específico para 5K (intervalados curtos em pace I). O 5K é corrido próximo ao VO2max — o treino precisa refletir isso. Use o pace atual como referência e projete metas alcançáveis."
- **Additional questions:** Nenhuma
- **Remove questions:** Nenhuma
- **Metrics override:** VDOT completo, zonas de treino, projeção de 10K via Riegel, percentil brasileiro
- **Narrative arc override:** "Seu pace hoje → o que seu VDOT revela → onde está o gargalo → como a polarização desbloqueia → seus paces de treino por zona → próximos passos com o profissional"

**Justificativa:** O evoluidor de 5K é o público-core deste template. Máxima profundidade técnica, máximo uso de métricas calculadas, máximo valor do VDOT. É o relatório que mais impressiona e mais converte — porque entrega algo que o corredor não consegue sozinho.

---

### Branch: `Evoluidor 10K` (trigger: `race_focus == improve_10k`)

- **Tom geral:** Estratégico. 10K exige gestão de esforço, periodização, e uma base aeróbica mais robusta que 5K. Tom de "construção" — blocos de preparação, fases de treino. Pode usar conceitos de periodização (base, construção, específico, polimento).
- **pillarGuidance:** "Foque em construção de volume aeróbico, corrida tempo/limiar, e gestão de prova (negative split, hidratação). O 10K é corrido a ~90% do VO2max — precisa de base mais sólida que o 5K. Inclua orientação sobre longão progressivo como sessão-chave. Se o corredor já corre 5K, projete tempo de 10K e use como meta."
- **Additional questions:** Nenhuma
- **Remove questions:** Nenhuma
- **Metrics override:** VDOT completo, zonas de treino (ênfase em zona T/Tempo), projeção 5K → 10K via Riegel, projeção 10K → Meia via Riegel, percentil brasileiro
- **Narrative arc override:** "Sua base atual → o que o 10K exige vs 5K → seu potencial estimado → como construir volume com segurança → gestão de prova → próximos passos com o profissional"

**Justificativa:** A diferença entre 5K e 10K não é só distância — é fisiologia e estratégia. No 5K, VO2max domina. No 10K, limiar anaeróbio e economia de corrida ganham peso. O relatório precisa refletir essa diferença nas recomendações e no arco narrativo. Pilares de "gestão de prova" (hidratação, negative split, estratégia de ritmo) só fazem sentido para 10K+ — em 5K são secundários.

## 7 · Métricas para o ALUNO (relatório)

### 7.1 · `estimated_vdot`

- **O que é:** VDOT estimado a partir do pace recente informado, usando a fórmula de Jack Daniels
- **Por que aparece para o aluno:** É a "nota" do corredor — um número único que resume onde ele está. Engajante porque é comparável ("meu VDOT é 38, quero chegar a 42") e porque desbloqueia tudo o que vem depois (zonas, projeções)
- **Cálculo:** VDOT = oxygen cost / sustainable %VO2max, usando pace médio convertido para velocidade (m/min) e distância assumida (5K para pace < 6:00/km, 3K genérico para paces mais altos). Ref: Daniels, _Running Formula_
- **Visualização proposta:** Card numérico grande com valor central + badge de classificação (Iniciante / Intermediário / Avançado / Elite). Rationale: VDOT é um número único — não precisa de gauge, precisa de contexto qualitativo
- **Placeholder:** `[vdot_score]`, `[vdot_class]`

### 7.2 · `training_zones`

- **O que é:** Paces de treino derivados do VDOT (Easy, Marathon, Tempo, Interval, Repetition) em formato mm:ss/km
- **Por que aparece para o aluno:** É a entrega mais acionável do relatório. O corredor sai sabendo EXATAMENTE em que pace correr cada tipo de sessão. Resolve o gargalo #1 (correr tudo na mesma intensidade)
- **Cálculo:** Tabelas de Daniels — cada VDOT mapeia para faixas de pace por zona. Ref: Daniels, _Running Formula_, Table 3.1
- **Visualização proposta:** Zone table com 5 linhas (E/M/T/I/R), cada uma com nome em PT-BR, faixa de pace, e cor semafórica (verde → amarelo → vermelho). Rationale: tabela de zonas é o formato padrão que corredores reconhecem e consultam antes de cada treino
- **Placeholders:** `[pace_easy]`, `[pace_tempo]`, `[pace_interval]`, `[pace_rep]`

### 7.3 · `race_prediction`

- **O que é:** Tempo estimado em 5K, 10K e Meia-maratona baseado no pace atual
- **Por que aparece para o aluno:** Projeção de potencial é profundamente engajante — "com sua base atual, você poderia fazer 10K em 52:30". Cria meta concreta e motiva
- **Cálculo:** Fórmula de Riegel: T2 = T1 × (D2/D1)^1.06. Ajuste: para corredores com VDOT < 35, usar expoente 1.08 (menos treinados perdem mais pace em distâncias maiores). Ref: Riegel, 1981
- **Visualização proposta:** 3 cards lado a lado (5K / 10K / 21K), cada um com tempo estimado e pace correspondente. Destaque na distância-alvo do branch. Rationale: formato visual limpo que permite comparação instantânea
- **Placeholders:** `[pred_5k]`, `[pred_10k]`, `[pred_21k]`

### 7.4 · `brazilian_percentile`

- **O que é:** Posição percentil do corredor entre amadores brasileiros do mesmo sexo e faixa etária
- **Por que aparece para o aluno:** Contextualiza performance — "seu pace te coloca entre os 40% mais rápidos da sua faixa". Muito mais significativo que um número absoluto. Valida quem é mais rápido do que pensa e motiva quem tem margem
- **Cálculo:** Tabelas de percentil por sexo e faixa etária de Pecanha et al. 2021 (MDPI Healthcare). Interpolação linear entre faixas publicadas
- **Visualização proposta:** Gauge horizontal com marcador de posição do corredor na distribuição (0-100%), com faixas coloridas (top 10% dourado, 10-30% verde, 30-60% azul, 60%+ cinza). Rationale: gauge posicional é intuitivo e emocionalmente impactante
- **Placeholder:** `[percentile]`, `[percentile_context]`

### 7.5 · `weekly_structure`

- **O que é:** Sugestão de estrutura semanal de treino baseada na frequência informada
- **Por que aparece para o aluno:** Traduz os conceitos técnicos em ação concreta — "na terça rodagem, na quinta intervalado, no sábado longão". O corredor sai do relatório sabendo o que fazer amanhã
- **Cálculo:** Não é cálculo — é template de distribuição baseado em frequência × branch. Regras: 80% volume em zona E, 20% em zonas T/I/R (princípio 80/20 de polarização, Seiler 2010)
- **Visualização proposta:** Grid semanal (dom-sáb) com ícones por tipo de sessão (rodagem, intervalado, longão, descanso, fortalecimento). Rationale: calendário visual é mais acionável que texto corrido
- **Placeholder:** `[weekly_plan]`

## 8 · Métricas para o PROFISSIONAL (dashboard futuro)

### 8.1 · `vdot_score`

- **O que é:** VDOT numérico bruto (sem arredondamento)
- **Por que importa para o PT:** Permite comparar leads entre si, estratificar turmas por nível, e acompanhar evolução em avaliações subsequentes
- **Cálculo:** Mesmo da seção 7.1
- **Visualização proposta:** Número com sparkline de evolução (se houver avaliações anteriores)

### 8.2 · `injury_risk_score`

- **O que é:** Score 0-10 de risco de lesão, derivado de: frequência alta + sedentarismo recente + lesão articular ativa + gargalo = injury_fear
- **Por que importa para o PT:** Flag de alerta — lead com score alto precisa de avaliação presencial antes de começar planilha
- **Cálculo:** Heurística ponderada: `joint_injury` (+4), `recent_sedentary` + `freq_4_5/6_plus` (+3), `injury_fear` (+2), `freq_6_plus` sem fortalecimento (+1)
- **Visualização proposta:** Semáforo (verde/amarelo/vermelho) com tooltip expandível mostrando fatores

### 8.3 · `adherence_prediction`

- **O que é:** Probabilidade estimada de adesão ao treino nas próximas 8 semanas
- **Por que importa para o PT:** Lead com baixa adesão prevista precisa de estratégia diferente (mais contato, sessões menores, accountability)
- **Cálculo:** Heurística: `freq_1` + `time_consistency` = adesão baixa; `freq_4_5` + `pace_plateau` = adesão alta (já tem hábito, quer resultado)
- **Visualização proposta:** Badge (Alta / Média / Baixa) com cor

### 8.4 · `training_age`

- **O que é:** Tempo estimado de prática regular de corrida (inferido de frequência + VDOT + gargalo)
- **Por que importa para o PT:** Corredor com VDOT 45 e 2x/semana = talento natural ou ex-atleta. Corredor com VDOT 30 e 5x/semana = precisa de técnica, não de volume. A idade de treino contextualiza
- **Cálculo:** Heurística cruzada (não há fórmula validada — anotar em Pendências)
- **Visualização proposta:** Badge (Novato < 6m / Intermediário 6m-2a / Experiente 2a+)

### 8.5 · `recommended_focus`

- **O que é:** Sugestão de foco principal para o plano de treino desse lead, baseada no cruzamento de gargalo + nível + branch
- **Por que importa para o PT:** Poupa tempo — ao abrir o lead no dashboard, já sabe se o foco é "base aeróbica", "limiar", "fortalecimento", ou "gestão de frequência"
- **Cálculo:** Regras de mapeamento: `pace_plateau` → "Polarização + limiar"; `injury_fear` → "Fortalecimento + volume controlado"; `no_structure` → "Periodização básica"; etc.
- **Visualização proposta:** Tag colorida com texto curto (ex: "Polarização", "Fortalecimento", "Base aeróbica")

## 9 · Pilares do relatório

### Pilar 1 · Treino Inteligente

- **Subtitle:** "Corra menos no esforço, mais no resultado"
- **Conceito central:** A maioria dos corredores intermediários treina 100% das sessões na mesma intensidade moderada — a "zona cinza". Polarização de treino (80% fácil, 20% forte) é o princípio mais validado pela ciência para melhorar pace. Correr devagar constrói base; correr forte empurra limiar. A zona do meio não faz nenhum dos dois bem.
- **Evidência científica:** Seiler, S. "What is best practice for training intensity and duration distribution in endurance athletes?" (_Int J Sports Physiol Perform_, 2010). Esterman, A. & Cooke, J. "Polarised training model for endurance running" (_J Sports Sci_, 2019).
- **Placeholders esperados:** `[pace_easy]`, `[pace_tempo]`, `[pace_interval]`, `[weekly_plan]`, `[profissional_nome]`
- **Exemplo de texto popular (80 palavras):** "A grande sacada dos corredores que quebram RP não é treinar mais — é treinar diferente. Seus treinos fáceis precisam ser REALMENTE fáceis: pace [pace_easy], respiração tranquila, conversando. Parece lento? É proposital. Essa base é o que permite que nos dias de tiro (pace [pace_interval]) você realmente force. Com [profissional_nome], você vai aprender a dosar cada sessão para que o conjunto te leve mais longe."
- **Exemplo de texto técnico (35 palavras):** "Modelo polarizado (80/20): ≥80% do volume semanal em zona E (pace [pace_easy]), ≤20% em zonas T/I/R. Evidência consistente de superioridade sobre modelo threshold para corredores amadores (Seiler, 2010)."

### Pilar 2 · Corpo Preparado

- **Subtitle:** "Corrida se constrói fora da corrida também"
- **Conceito central:** Lesões em corredores não vêm de "correr demais" — vêm de correr sem a estrutura muscular para absorver impacto. Fortalecimento de glúteo, core e panturrilha reduz incidência de lesão em 50%+. Cadência e técnica de pisada impactam economia de corrida.
- **Evidência científica:** Lauersen, J.B. et al. "The effectiveness of exercise interventions to prevent sports injuries: a systematic review and meta-analysis" (_Br J Sports Med_, 2014). Heiderscheit, B.C. et al. "Effects of step rate manipulation on joint mechanics during running" (_Med Sci Sports Exerc_, 2011).
- **Placeholders esperados:** `[biggest_barrier]`, `[training_frequency]`, `[profissional_nome]`
- **Exemplo de texto popular (75 palavras):** "Seu maior aliado na corrida pode não ser o tênis — são seus músculos de suporte. Glúteo que não ativa, core que não estabiliza, panturrilha que cansa antes dos pulmões. Duas sessões curtas de fortalecimento por semana (20min) reduzem pela metade o risco de lesão. Não é musculação pesada — são exercícios específicos para o gesto da corrida. [profissional_nome] pode montar essa base com você."
- **Exemplo de texto técnico (30 palavras):** "Fortalecimento neuromuscular 2x/semana (glúteo médio, core anti-rotação, panturrilha excêntrica) reduz risco de lesão em 50% (Lauersen et al., 2014). Complemento não-negociável para volume >30km/sem."

### Pilar 3 · Estratégia de Prova

- **Subtitle:** "No dia que importa, cada decisão conta"
- **Conceito central:** A diferença entre uma boa prova e uma frustração raramente é condicionamento — é gestão. Sair rápido demais nos primeiros km, não hidratar, não testar o pace de prova em treino. Negative split (segunda metade mais rápida) é a estratégia mais eficiente e menos utilizada por amadores.
- **Evidência científica:** Abbiss, C.R. & Laursen, P.B. "Describing and understanding pacing strategies during athletic competition" (_Sports Med_, 2008). Santos et al. "Pacing strategy in running" (_Int J Sports Med_, 2015).
- **Placeholders esperados:** `[pred_5k]`, `[pred_10k]`, `[race_focus]`, `[pace_tempo]`, `[profissional_nome]`
- **Exemplo de texto popular (70 palavras):** "Sabe aquele km 1 em que você sai voando e nos últimos 2 quase para? Todo corredor já viveu. A estratégia que os melhores usam é o oposto: começar contido e acelerar no final. Seu pace de largada ideal é [pace_tempo] — confortavelmente desconfortável, sem ofegar. Guarde energia para quando os outros desaceleram. [profissional_nome] vai te preparar para executar isso."
- **Exemplo de texto técnico (30 palavras):** "Negative split: primeiros 60% da prova a 95% do pace-alvo, últimos 40% a 100-103%. Reduz percepção de esforço global e otimiza distribuição de glicogênio (Abbiss & Laursen, 2008)."

## 12 · Notas de design (decisões não-óbvias)

### Por que NÃO incluí pergunta sobre frequência cardíaca máxima

Muitos templates de corrida pedem FC máxima ou FC de repouso. Descartei porque: (a) a maioria dos corredores não sabe sua FC máxima real (fórmulas 220-idade são imprecisas ±10-12 bpm), (b) FC de repouso requer medição matinal que poucos fazem, (c) para 5K/10K, zonas por pace (derivadas do VDOT) são mais acionáveis que zonas por FC. Se o corredor tem relógio com FC, o PT configura zonas depois presencialmente.

### Por que `recent_pace` é optional mesmo quando visível

Muitos corredores usam GPS mas não lembram o pace da última corrida, ou fizeram uma corrida atípica (muito devagar, muito rápido). Forçar resposta geraria dados imprecisos. Se não preencher, o relatório funciona sem VDOT (usa recomendações genéricas por branch) — perde profundidade mas não quebra.

### Por que NÃO pergunto distância do longão ou volume semanal

Volume semanal total e distância do longão são dados valiosos para o PT, mas exigem que o corredor calcule ("quantos km eu fiz no total semana passada?"). Viola a regra dos 5-10 segundos para responder. Frequência semanal é proxy suficiente — corredor que treina 4x/semana com pace 5:30 tem perfil de volume razoavelmente previsível.

### Por que o branch "Primeira Prova" agrupa `first_5k` + `first_race`

Ambos os perfis compartilham a mesma necessidade: relatório focado em experiência (não performance), sem métricas intimidadoras, tom de "preparação para o dia". A diferença entre "quer fazer 5K" e "quer competir mas não sabe a distância" não justifica branches separados — o arco narrativo é o mesmo.

### Por que `nutrition_doubt` é gargalo mas não gera branch

Nutrição é gargalo real, mas o relatório de corrida não é o lugar para prescrição alimentar (escopo de nutricionista). O gargalo serve para calibrar o pilar 1 (mencionar timing de carb) e o tom geral (não sugerir "corra em jejum"), mas não muda métricas ou estrutura. Se virasse branch, prometeria algo que o template não pode entregar.

### Por que mantive o label "5K / 10K" e não separei em 2 templates

Considerei criar templates separados (um para 5K, outro para 10K). Descartei porque: (a) o público se sobrepõe enormemente — quem corre 5K geralmente mira 10K em seguida, (b) os branches + arco narrativo já diferenciam suficientemente, (c) 2 templates separados duplicariam 80% do conteúdo sem ganho proporcional. A projeção Riegel conecta as duas distâncias naturalmente.

### Sobre o label proposto

Mantenho `rua_5k_10k` como specialty_code e sugiro **"Corrida de Rua — 5K / 10K"** como label visível no hub. "Corrida de Rua" é identitário; "5K / 10K" qualifica. Se o label for longo demais para o card, pode abreviar para "5K / 10K" com subtítulo "Corrida de Rua".

## 13 · Pendências

1. **VDOT a partir de pace de treino vs pace de prova:** O VDOT de Daniels é calibrado para performance de corrida all-out em prova. Pace médio de treino ≠ pace de prova. Preciso definir fator de correção (estimado: 5-8% mais rápido em prova) ou buscar tabela auxiliar que aceite pace de treino como input. → **Decisão necessária antes da implementação**

2. **Tabelas de percentil brasileiras (Pecanha 2021):** O estudo publica percentis por sexo e faixa etária para 5K, 10K, 21K e 42K. Preciso extrair as tabelas numéricas para implementar interpolação. → **Dados disponíveis no paper, falta digitalizar**

3. **Heurística de `training_age` e `adherence_prediction`:** Não há fórmula validada — são scores heurísticos baseados em cruzamento de respostas. Precisam de calibração com dados reais de assessorias. → **Aceitável como v1 heurística, refinar com dados**

4. **Lib para cálculo VDOT:** Identificar se existe lib npm/TS que implementa as tabelas de Daniels. Se não, implementar as fórmulas do livro (são públicas). → **Pesquisa de lib pendente**

5. **Cadência como métrica:** Cadência (passos/min) é métrica relevante mas não é coletada no formulário (exigiria pergunta extra ou integração com Strava). Documentada aqui para feature futura de integração com wearables. → **Adiada para v2**

6. **PSE como fallback para quem não tem GPS:** Para corredores sem GPS (`no_gps`), o relatório perde as métricas de VDOT/zonas/projeção. Preciso garantir que os pilares funcionem sem essas métricas — foco em estrutura qualitativa (tipo de sessão por PSE, não por pace). → **Testar com IA que os pilares funcionam nos 2 cenários**
